# OTel与日志采集：数据流水线架构 (Data Pipeline)

为了打造一个极致透明化的 AI 运行可观测平台，必须跨平台、跨代码逻辑地去捕获任何的工具使用痕迹和状态，这无疑是对数据链路（Data Pipeline）极大的考验。

OpenClaw 采取了类似云原生应用常用的标准设计（以 **OTel [OpenTelemetry]** 为规范思路），实现了完全无代码侵入的闭环日志接入机制。

---

## 🛠️ 流水线整体设计

```text
┌─────────┐    ┌───────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│OpenClaw │───►│ Vector Pipeline   │───►│ Apache Doris    │◄───│    Frontend     │
│ Agent   │    │ (Data Collection  │    │ (Storage &      │    │ (Visualization) │
│ Logs    │    │  & Transformation)│    │  Analytics)     │    │   React / Vite  │
└─────────┘    └───────────────────┘    └─────────────────┘    └────────┬────────┘
                                                                        │
                                           ┌─────────────────┐          │
                                           │   Backend API   │◄─────────┘
                                           │   (Node.js)     │
                                           │   Port: 8787    │
                                           └─────────────────┘
```

该架构中涉及以下几个重点流转环节：

### 1. 源头注入与导出 (Source & Logs Generation)
当你运行诸如基于 KWeaver Core 或使用底层 eBPF 探针建立的数字化员工节点时，业务应用并不需要“主动”呼叫网络上的分析接口（这往往会引发同步延迟）。
Agent 在每一次的 prompt 发送、结果拉取以及环境 I/O 读取中，仅仅会在宿主机本地产生结构化的日志文件（类似于常用的 *.jsonl 或是 session.json 对象映射）。

> 此做法完美隔离了监控模块和核心业务代码：**即使采集平台崩溃，AI 员工也能正常工作**。

### 2. 遥测过滤与变换网关 (Vector Transform)
对于每一批产生变动的本地物理文件游标，我们会拉起一个超轻量的开源分析器 `Vector` 服务作为数据分发守护进程。根据已配置的 `vector.yaml`：
- **Source 读取**：实时拉取和监听对应文件夹内尾部新增的数据；
- **Transform (Remap / Reduce)**：运用内置的处理脚本将庞大、无序但深嵌套的 JSON `payload` 摊平，提取出诸如 `.log_attributes`（整个日志本身）和作为主键的 `.session_id` 等索引标签；
- **清理无用信息**：直接将部分系统垃圾字段予以清理不进行上报。

### 3. 数据高吞吐入库核心 (Apache Doris)
在所有数据清洗结束后，采用 `HTTP Sinks` 分批将 JSONL 按行高速敲入 Doris OLAP 数据库的接收队列中：
- 依靠 `_stream_load` 协议的极高并发能力，平台可在千万行级别实现秒级落库；
- 数据被装载进诸如 `agent_sessions_logs` 等明细表和索引表中待查。

## 🔎 组件性能兼容说明

我们选取该日志汇聚方式的原因在于它极高的稳定性和行业普适性：
1. **轻量与隔离**：哪怕目标机器上跑了上百个并行的容器化智能体，Vector 以不到几十兆的内存资源占用就能搞定所有的采集上报。
2. **扩缩容非常容易**：对于后续海量的增长，后端可以直接增加 Doris BE 节点，轻松获得几倍于从前的查询承载与汇聚能力！
