# 特权与配置变更审计 (Config Audit)

在数字化员工（AI Agent）拥有一定的系统级工具调用权限（如 Shell 执行、文件读写 API）时，如何确保基础机器和生产环境不被恶意指令（或者大模型的幻觉输出）损坏？这是基础设施合规性的最后一道守护者。

配置变更审计不仅关注 Agent 回答了什么文本，更关注它在操作系统后台搞了什么破坏或发生了何种越权。

---

## 🌟 核心价值

让系统保持“状态可追溯可逆转”。当某台 Agent 服务器出现网络不通、进程意外消失、挂载盘被清除等运维问题时，不再需要让工程师去费劲扒取机器命令历史，所有敏感修改都将作为证据呈现在审计系统面板上。

## 🛡️ 主力拦截与追溯维度

### 1. 追踪高危系统文件篡改

基于 OTel 或探针级别设定的监控点，当 Agent 执行环境中有动作触碰到类似如下的系统核心路径，都将产生一次 Audit 审计记录抛送到我们的 Apache Doris 服务进行封存：
- 系统网络配置：诸如意外写入或强行覆写了 `/etc/hosts`、`/etc/resolv.conf` 等；
- 私钥证书与密钥库：如试图窃取或更新 `~/.ssh/authorized_keys`；
- 环境变量脚本：如 `~/.bashrc` 或 `/etc/profile`；
- Kubernetes Config 与数据库密码挂载卷等。

> **如何配置特定阻断 (How to Filter) 👉** Agent 的 `Tool` 实现层中（比如 OpenClaw 的 `CommandExecutor`）会自动把每一次写文件的 `dest` 地址和读文件的 `path` 进行识别报告，只要你的应用规范了这个字段的发送即可。

### 2. 执行前后的状态差异 (Diff Analysis)

如果数字员工出于修补某个缺陷的目的，获得了操作系统的正式批准去修改 Nginx 或者是某种程序的 `yaml`/`json` 配置文件。平台依然会自动把：
- **覆写前的原始内容**
- **以及写入后的新文本或补丁 (Patch)**

...等参数统统抓留下来，并直观展现为差异化比对面板（Diff Viewer），随时方便你使用类似 `git diff` 一样的体验，将一次出格操作对环境造成的漂移一目了然！
