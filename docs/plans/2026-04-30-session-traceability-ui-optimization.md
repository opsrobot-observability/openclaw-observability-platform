# 会话链路溯源 UI 优化计划

> **给 Claude：** 必须使用子技能：`superpowers:executing-plans`，并按任务逐步执行本计划。

**目标：** 优化“会话链路溯源”列表页与“会话详情”体验，让安全审计人员能够快速分诊高风险会话，并按时间顺序重建事件证据链。

**架构：** 保持现有 React 页面结构、国际化文件、Tailwind 工具类样式，以及 `/api/agent-sessions`、`/api/agent-sessions-logs` 接口契约不变。展示层重组为两段式体验：列表页以风险优先级为核心，详情页以证据驱动的取证叙事为核心。除非实现过程中发现现有映射行与 JSONL 解析辅助函数无法推导所需数据，否则避免修改后端数据结构。

**技术栈：** React、Vite、Tailwind CSS、`react-intl-universal`，以及 `frontend/lib/sessionAudit.js` 中现有的会话审计辅助函数。

---

## 产品诊断

### 1. 会话链路溯源列表

当前列表更像一张原始数据表，而不是安全审计工作台。它默认展示 17 列，最小宽度达到 1880px，导致用户必须横向扫描，也让最关键的问题变得不够突出：**“哪些会话需要优先审计？”** 风险数据虽然已经存在，但只是众多列中的一列。列表还把 provider、chat type、channel、token 数、文件操作数、执行次数等元数据赋予了几乎相同的视觉权重，而这些字段在审计决策中的价值并不相同。

从“风险感知”的安全审核视角看，列表应优先呈现：

1. 风险等级与数量：高危、中危、低危。
2. 可疑行为信号：网络访问、文件操作、exec/process 行为、异常中断。
3. 调查身份信息：会话 ID、Agent、时间窗口、会话标签。
4. 成本/模型元数据：模型、Token 用量、Provider。
5. 次级上下文：Channel、Chat Type、Origin Provider。

### 2. 会话详情

当前详情页包含很多有价值的证据，但布局是按功能标签组织的：Trace Analysis、Chat Details、Intent Recognition、Model Invocation、Tool Invocation、Network & Files、Risk Perception。这个结构从技术模块角度是合理的，但不符合事后审计取证的工作方式。审计人员需要先建立上下文，判断哪里出了问题，再查看时间线，将高风险行为与原始证据关联起来，最后复制或导出精确证据。

详情页应转为一份调查取证档案：

1. 案件摘要：谁、做了什么、何时发生、结果如何、最高风险、风险原因。
2. 证据时间线：按时间排序的事件，带风险标记、时间间隔与来源行锚点。
3. 风险发现：按优先级排列，并可跳转到时间线中的对应证据。
4. 行为转写：对话、工具、模型、网络证据作为关联支撑材料。
5. 原始凭证：可展开的 JSON 与复制操作保留，但不作为默认视觉主体。

## 推荐设计方向

采用 **“风险分诊 -> 证据档案”** 模型。

列表页应成为风险分诊队列。默认视图减少列数，强化风险排序，并提供清晰的调查入口。密集的运维元数据应移入紧凑徽章、次级详情展开区，或后续可选的“技术列”模式中。

详情页应成为取证案件页。顶部区域先总结案件与风险态势。主体区域以时间线为锚点，风险发现与各类证据围绕时间线展开，而不是作为互相竞争的平级标签存在。现有按数据类型划分的标签可以保留，但应被重新定义为证据筛选或次级面板。

已考虑的替代方案：只隐藏部分列并保留当前标签式详情页。这个方案成本较低，但无法解决审计逻辑不清的问题。另一个方案是为调查档案创建全新路由，长期可能有价值，但对本 issue 来说范围过大，因为现有 `SessionAudit.jsx` 已经包含所需数据和下钻流程。

## 目标信息架构

### 列表页：默认安全队列

默认可见列：

1. 风险：最高风险等级、高/中/低风险数量、异常中断标记。
2. 会话：会话 ID、session key 预览、标签。
3. 行为主体：Agent 名称与 Channel。
4. 时间窗口：开始时间与持续时长。
5. 关键证据：网络、文件、exec、工具调用的紧凑徽章。
6. 模型与成本：模型与总 Token，视觉上作为次级信息。
7. 操作：打开调查。

默认隐藏或移入展开详情的信息：

- 结束时间。
- Chat Type。
- Origin Provider。
- Provider。
- Channel 的原始变体字段。
- 已在“关键证据”中汇总的 network/file/exec 独立计数列。

默认排序：

1. 高危数量降序。
2. 中危数量降序。
3. 异常中断会话优先。
4. 最近更新时间或开始时间优先。

推荐筛选：

- 风险等级分段控制：全部、高危、中危、低危、清洁。
- 证据开关：网络、文件、Exec、异常中断。
- 搜索保持全局搜索，但 placeholder 应提示可搜索的审计对象。

### 详情页：调查取证档案

顶部摘要：

- 案件标题行：会话 ID、Agent、最高风险徽章、状态/结果。
- 时间范围、持续时长、总行数、可解析比例。
- 高/中/低风险数量。
- 关键证据计数：工具调用、网络 URL、文件写入、exec/process 事件。
- 复制会话 ID 操作保持可见。

主体布局：

- 左侧/主列：按时间排序的证据时间线。
- 右侧/桌面端吸附列：风险发现、证据计数、跳转链接。
- 移动端使用单列堆叠布局。

证据导航：

- 将当前同等权重的标签替换为面向审计的分区：
  - 概览
  - 时间线
  - 风险发现
  - 对话
  - 工具与模型
  - 网络与文件
  - 原始证据
- 每条风险发现都应能跳转到时间线中的来源行。
- 时间线事件应暴露相关的对话、工具、模型、网络证据，减少用户在多个标签之间切换上下文的成本。

## 交互设计细节

### 列表页

- 使用适合运维工具的行密度：紧凑，但不要像电子表格一样拥挤。
- 将“风险”作为第一列，并使用清晰的徽章：高危、中危、低危、清洁。
- 将 `H 0 · M 0 · L 0` 这种纯文本替换为紧凑的风险堆叠与颜色编码等级。
- 将 network/file/exec/tool 计数合并到一个“证据”单元格中，用小徽章表示。
- 在行尾提供“打开调查”按钮或图标按钮，同时保留整行点击与键盘激活能力。
- 只有在用户明确需要原始表格行为时，再考虑增加列密度控制；第一阶段不实现。
- 横向滚动可以作为兜底，但默认表格应能适配常见桌面宽度，不应要求用户横向滚动才能完成核心审计。

### 详情页

- 首屏应回答：“这个会话是否有风险？为什么？我应该先看哪里？”
- 将风险数量与来源证据计数移到 transcript analysis 标签之前。
- 当存在高危/中危风险时，展示“风险发现优先”的面板。
- 保留时间线回放，但将其作为次级控件。回放有价值，但事后审计人员通常更需要定点跳转，而不是动画播放。
- 增加持久跳转入口：时间线、风险发现、对话、工具、网络、原始证据。
- 原始 JSON 作为证据凭证保留，但通过明确的展开与复制操作访问。
- 在风险发现、时间线、对话、工具、原始 JSON 中统一使用行号作为证据锚点。

---

## 视觉设计规则

- 延续现有卡片、边框与 Tailwind 设计语言。
- 避免装饰性看板或营销式 hero 区块；这是审计运维界面。
- 使用克制配色，红/琥珀/天蓝/翠绿仅用于风险与健康状态。
- 密集运维控件尽量使用 8px 或更小圆角；现有 `app-card` 圆角可为了一致性保留。
- 如果项目已有图标库，操作按钮优先使用图标；否则使用紧凑文本按钮，避免自定义 SVG 泛滥。
- 改善键盘用户的 focus 状态，尤其是行、标签、跳转链接和可展开证据项。
- 风险徽章与弱化元数据应满足 WCAG AA 对比度要求。

## 验收标准

- 默认“会话链路溯源”列表能够适配常见桌面宽度，用户无需为了完成核心审计工作而强制横向扫描。
- 风险优先级在列表中同时具备视觉优先级与功能优先级。
- 列表支持通过风险与证据信号更快完成分诊。
- “会话详情”页读起来像一份取证调查档案，而不是一组彼此独立的功能模块。
- 风险发现可以直接链接到时间线证据。
- 原始 JSON 证据仍可用于证明与校验。
- 加载、空数据、错误、密集数据、移动端、深色模式、键盘操作等状态保持精致可用。
